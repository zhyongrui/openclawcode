## Bounded Builder And Verifier Timeouts

The next refreshed-branch live proof exposed a new blocker distinct from the
earlier provider `HTTP 400` failures.

- proof target:
  - issue `#85`
- proof state dir:
  - `.openclawcode-proof-sync-20260313`
- observed live symptom before the fix:
  - run `zhyongrui-openclawcode-85-1773414881879` stalled in `building`
  - saved history stopped at `Build started`
  - no bounded failed artifact surfaced until the outer proof wrapper timed out

Diagnosis:

- the embedded runner already supported `timeoutSeconds`
- `AgentBackedBuilder` and `AgentBackedVerifier` were not forwarding that field
  into `agentRunner.run(...)`
- `openclaw code run` was also not setting workflow-specific timeout defaults,
  so live proofs inherited the host's broader generic agent timeout

Implemented fix:

- `AgentBackedBuilder` now forwards `timeoutSeconds`
- `AgentBackedVerifier` now forwards `timeoutSeconds`
- `openclaw code run` now sets workflow defaults:
  - builder: `300` seconds
  - verifier: `180` seconds
- operator overrides are now available through:
  - `OPENCLAWCODE_BUILDER_TIMEOUT_SECONDS`
  - `OPENCLAWCODE_VERIFIER_TIMEOUT_SECONDS`
- generic non-provider workflow failures now also persist
  `failureDiagnostics.summary`, so timeout-style failures remain visible in
  structured workflow artifacts and operator status

Validation:

- `pnpm exec vitest run src/openclawcode/roles/agent-backed.test.ts src/commands/openclawcode.test.ts src/openclawcode/testing/run-issue.test.ts --pool threads`
- `pnpm exec vitest run --config vitest.openclawcode.config.mjs --pool threads --maxWorkers 1`
- `pnpm build`

Additional test-harness hardening during this slice:

- widened the explicit timeout budget for two known heavy full-suite cases:
  - `src/openclawcode/testing/run-issue.test.ts`
  - `src/openclawcode/testing/orchestrator.test.ts`

## Setup-Check Runtime Selection Hardening

The next live-ops proof exposed a smaller but real operator-preflight gap.

- direct observation:
  - `./scripts/openclawcode-setup-check.sh --strict --json` could still drag on
    while probing `dist/index.js models list --json`
  - the root cause was that the script already knew PATH `node` was below the
    CLI startup floor on this host, but it still tried to run the built CLI
    with that same runtime

That is now fixed:

- `scripts/openclawcode-setup-check.sh` now accepts:
  - `OPENCLAWCODE_SETUP_NODE_BIN=/path/to/node>=22.16.0`
  - `OPENCLAWCODE_SETUP_CLI_PROBE_TIMEOUT_SECONDS=<seconds>`
- direct CLI probes now run through one bounded helper
- if the selected Node runtime is already below the CLI startup floor, the
  script reports the failure and skips model-inventory probing instead of
  hanging in the CLI
