# 2026-03-14

## Built OpenClawCode Plugin Startup Fix

Implemented the built-runtime startup repair for the bundled `openclawcode`
plugin.

- `tsdown.config.ts`
  - now emits `dist/extensions/openclawcode/index.js`
- `scripts/copy-extension-manifests.ts`
  - now copies bundled `openclaw.plugin.json` files into `dist/extensions/*`
- `package.json`
  - build scripts now run `copy-extension-manifests.ts`
- `src/plugins/bundled-dir.ts`
  - now prefers `<packageRoot>/extensions` before ancestor walking so a partial
    `dist/extensions` tree does not shadow the full bundled plugin set
- `src/plugins/discovery.ts`
  - now supports selective built bundled overrides
  - built runtime, or `OPENCLAW_PREFER_BUILT_BUNDLED_PLUGINS=1`, now redirects
    bundled `openclawcode` to `dist/extensions/openclawcode/index.js`
- `src/plugins/discovery.test.ts`
  - now covers the built bundled override path
- `src/plugins/bundled-dir.test.ts`
  - now covers the package-root `extensions/` preference over `dist/extensions`

Field note:

- a naive "openclawcode-only" diagnostic config is still too broad
- bundled defaults such as `device-pair`, `ollama`, `phone-control`, `sglang`,
  `talk-voice`, `vllm`, and `memory-core` still enable themselves unless the
  proof config also sets:
  - `plugins.allow = ["openclawcode"]`
  - `plugins.slots.memory = "none"`

Validated locally with:

- `pnpm exec vitest run src/plugins/bundled-dir.test.ts --pool threads`
- `pnpm exec vitest run src/plugins/discovery.test.ts`
- `pnpm exec vitest run src/gateway/server-plugins.test.ts --pool threads`
- `pnpm exec vitest run --config vitest.openclawcode.config.mjs --pool threads --maxWorkers 1`
- `pnpm build`

Real proofs:

- bundled discovery now resolves `openclawcode` to:
  - `dist/extensions/openclawcode/index.js`
  - `dist/extensions/openclawcode/openclaw.plugin.json`
- direct `jiti()` of the compiled bundled `openclawcode` entry now completes
- direct loader proof with an allowlisted config now loads only `openclawcode`
  and finishes in about `2.3s`
- direct built gateway proof with that same allowlisted config now reaches:
  - `listening on ws://127.0.0.1:18890, ws://[::1]:18890`
  - `Browser control listening on http://127.0.0.1:18892/`

Next slice:

  component instead of reopening the bundled `openclawcode` loader path

## Setup Check Built Startup Proof

Turned the manual built-startup isolation proof into a repeatable setup-check
capability.

- `scripts/openclawcode-setup-check.sh`
  - now accepts `--probe-built-startup`
  - synthesizes an isolated proof config from the active operator config:
    - `channels = {}`
    - `bindings = []`
    - `plugins.allow = ["openclawcode"]`
    - `plugins.slots.memory = "none"`
  - runs the built `dist/index.js gateway run` entrypoint against that
    synthesized config on a non-default proof port
  - passes only when the listener line is observed
- `src/openclawcode/testing/setup-check.test.ts`
  - now covers the new startup-proof shell path with a stubbed built CLI

Validated locally with:

- `pnpm exec vitest run src/openclawcode/testing/setup-check.test.ts --pool threads`
- `pnpm exec vitest run --config vitest.openclawcode.config.mjs --pool threads --maxWorkers 1`
- `pnpm build`

Real host proof:

- `./scripts/openclawcode-setup-check.sh --strict --probe-built-startup --json`
  on `~/.openclaw` now returns:
  - `readiness.gatewayReachable = false`
  - `readiness.routeProbeReady = false`
  - `readiness.builtStartupProofRequested = true`
  - `readiness.builtStartupProofReady = true`
  - `readiness.nextAction = "start-or-restart-live-gateway"`

Live recovery follow-up:

- after restarting the long-lived repo-local gateway on commit `1a9aa38777`,
  `./scripts/openclawcode-setup-check.sh --strict --json` on `~/.openclaw`
  returned:
  - `summary.pass = 17`
  - `summary.warn = 1`
  - `summary.fail = 0`
  - `readiness.gatewayReachable = true`
  - `readiness.routeProbeReady = true`
  - `readiness.nextAction = "resolve-warnings-before-promotion"`
- the remaining strict blocker on the live host is now only the model-inventory
  warning, not gateway reachability or webhook route health

Model-inventory repair follow-up:

- `check_model_inventory()` now runs `models list --json` against a synthesized
  probe config with:
  - `channels = {}`
  - `bindings = []`
  - `plugins.enabled = false`
  - `plugins.entries = {}`
- that probe now passes on the real `~/.openclaw` host too:
  - `summary.pass = 19`
  - `summary.warn = 0`
  - `summary.fail = 0`
  - `modelInventory.available = 1`
  - `readiness.strict = true`
  - `readiness.nextAction = "ready-for-low-risk-proof"`

Field note:

- the allowlist and `plugins.slots.memory = "none"` are now codified in the
  proof path instead of living only in ad hoc debug notes, which makes
  promotion preflight repeatable for later operator hosts and future AI
  sessions
- real host validation against `~/.openclaw` returned:
  - `built gateway startup proof reached listener on ws://127.0.0.1:18890`
  - but overall `--strict --probe-built-startup --json` still failed with
    `gateway not reachable` and `signed webhook probe failed`
  - that means the isolated built-startup proof can now pass independently of
    the long-lived live gateway route, and operators should use
    `--skip-route-probe --probe-built-startup` when they are validating startup
    during a restart window

## Setup Check Readiness Split

Extended setup-check readiness so future sessions and rollout automation can
separate live-route failure from built-startup failure without scraping raw
check lines.

- `scripts/openclawcode-setup-check.sh`
  - readiness JSON now includes:
    - `gatewayReachable`
    - `routeProbeReady`
    - `routeProbeSkipped`
    - `builtStartupProofRequested`
    - `builtStartupProofReady`
  - when the isolated startup proof passes but the long-lived live gateway is
    still down, `nextAction` now becomes:
    - `start-or-restart-live-gateway`
- `src/openclawcode/testing/setup-check.test.ts`
  - now covers the split-readiness case explicitly
  - test helper timeout is widened to `30000` ms because the shell-heavy
    setup-check suite now has several legitimate slow paths

- `pnpm exec vitest run --config vitest.openclawcode.config.mjs --pool threads --maxWorkers 1`
