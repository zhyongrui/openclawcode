## Temporary Public Ingress Automation

Added a small operator-facing workflow for the current constraint where
`openclawcode` cannot use a truly fixed public domain yet and must rely on
anonymous `trycloudflare` URLs.

Implemented:

- `scripts/openclawcode-webhook-tunnel.sh`
  - `start`
    - starts a `cloudflared` quick tunnel
    - waits for the `https://*.trycloudflare.com` URL
    - updates the GitHub webhook URL automatically
  - `run`
    - runs the tunnel in the foreground for dedicated terminals or `tmux`
  - `sync-hook`
    - rewrites the GitHub webhook URL to the current tunnel host
  - `status`, `stop`, `restart`, `print-url`
- `docs/openclawcode/webhook-operations.md`
  - runbook for temporary public ingress mode
  - startup order and operator commands
- `docs/openclawcode/README.md`
  - linked the new webhook operations runbook from the index

Hardening details:

- the script now auto-sources `~/.openclaw/openclawcode.env` by default
- webhook sync writes both:
  - the current quick-tunnel URL
  - the local GitHub webhook secret, when present
- this avoids a regression where rewriting only the URL caused real GitHub
  deliveries to fail signature verification with `401`

Real validation:

- started the local gateway again on port `18789`
- established a new quick tunnel at:
  - `https://portsmouth-securities-creatures-meal.trycloudflare.com`
- updated GitHub webhook `600049842` for
  `zhyongrui/openclawcode` to the new tunnel URL
- replayed the real issue webhook delivery for issue `#40`
- GitHub reported the replay as:
  - `status: OK`
  - `status_code: 202`
  - delivered at `2026-03-11T02:31:02Z`

Operational note:

- anonymous `trycloudflare` hostnames are still ephemeral
- the new script removes the manual GitHub webhook editing step, but not the
  temporary-domain limitation itself
- issue `#40` currently has stale local workflow state from earlier interrupted
  runs, so the next clean end-to-end workflow validation should use a fresh real
  GitHub issue rather than reusing `#40`

## Webhook Runtime Hardening

Closed two runtime gaps that only showed up when replaying real GitHub issue
webhook deliveries against the plugin route.

Implemented:

- normalized the GitHub repository owner field in
  `src/integrations/openclaw-plugin/chatops.ts`
  - real GitHub issue payloads can send `repository.owner` as an object with a
    `login` field instead of a plain string
  - added `readGitHubRepositoryOwner(...)`
  - repo matching now accepts both owner shapes
- hardened `extensions/openclawcode/index.ts`
  - webhook intake now fails closed with `invalid-repository-owner` when the
    owner shape is unreadable
  - issue-webhook notifications now use `scheduleNotification(...)` so the
    webhook response can return without waiting on outbound chat delivery

Why this mattered:

- the real webhook payload for issue `#40` previously failed because the plugin
  assumed `repository.owner` was always a string
- awaiting chat delivery inside the webhook route risked making GitHub observe
  timeouts even after local issue intake had already succeeded

Added coverage in `extensions/openclawcode/index.test.ts`:

- accepts the real GitHub `repository.owner` object shape
- returns webhook acceptance without waiting for chat notification delivery

Validated locally with:

- `pnpm exec vitest run extensions/openclawcode/index.test.ts --pool threads`

## Operator Inbox Command: `/occode-inbox`

Added a small operator-facing inbox view for the bundled OpenClaw plugin so the
chat side can show pending approvals and recent workflow activity without
opening the raw plugin state file.

Implemented in `extensions/openclawcode/index.ts`:

- new `/occode-inbox` command
- optional repo argument:
  - `/occode-inbox owner/repo`
  - or `/occode-inbox` when exactly one repo is configured
- summarizes, for the selected repo:
  - pending approvals waiting on chat action
  - the current running issue
  - queued issue runs
  - the newest completed tracked runs from persisted snapshots

Why this matters:

- `approve` mode now has a direct chat-native operator overview
- the human no longer needs to remember exact issue numbers just to see what is
  waiting to be approved or what recently completed
- this is the first compact "run ledger" slice from the development plan,
  implemented without changing the workflow core

Added coverage in `extensions/openclawcode/index.test.ts`:

- `/occode-inbox` reports pending, running, queued, and recent-completed items
- `/occode-inbox` returns a compact empty summary when nothing is tracked
