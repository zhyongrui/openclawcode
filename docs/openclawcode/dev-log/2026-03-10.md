# Development Log - 2026-03-10

## Goal

Stabilize real GitHub issue execution so host-side validation commands can run
inside isolated issue worktrees without reinstalling dependencies for every run.

## Problem Observed

Real validation of issue `#4` proved that the workflow itself was behaving
correctly, but host-side test execution failed after the builder finished:

- the builder correctly detected that issue `#4` was already implemented
- the host then ran `pnpm exec vitest run --config vitest.openclawcode.config.mjs`
  inside the issue worktree
- that worktree had no local `node_modules`, so `pnpm exec vitest` failed with
  `Command "vitest" not found`

This was a worktree-host integration bug, not an issue-classification or prompt
targeting bug.

## Implementation

Added shared install artifact linking to the git worktree manager:

- `src/openclawcode/worktree/manager.ts` now defines
  `SHARED_INSTALL_ARTIFACTS = ["node_modules"]`
- when preparing a new or reused issue worktree, the manager now symlinks
  `repoRoot/node_modules` into the worktree if the source exists and the target
  does not
- changed-file collection now ignores shared install artifacts so the symlinked
  `node_modules` tree does not appear as workflow output noise

Added regression coverage in `src/openclawcode/testing/worktree.test.ts`:

- temp repos now include a fake `node_modules/.bin/vitest`
- new coverage proves the worktree receives a symbolic-link `node_modules`
  pointing back to the repository root
- changed-file reporting still excludes runtime/install artifacts

## Auto-Merge Validation And Hardening: Issues #13 And #15

Used GitHub issue `#13` to validate the next stage of the loop: real
`--merge-on-approve` behavior with a small command-layer feature
(`stageLabel` in `openclaw code run --json`).

First real `--merge-on-approve` validation:

- `node scripts/run-node.mjs code run --issue 13 --owner zhyongrui --repo openclawcode --repo-root /home/zyr/pros/openclawcode --branch-name openclawcode/issue-13-stage-label --builder-agent main --verifier-agent main --test "pnpm exec vitest run --config vitest.openclawcode.config.mjs" --open-pr --merge-on-approve --json`

Observed result:

- builder implemented the requested command-layer change and opened real draft
  PR `https://github.com/zhyongrui/openclawcode/pull/14`
- verifier approved the run for human review
- the final GitHub merge step failed with:
  `403 Resource not accessible by personal access token`

This exposed a workflow hardening gap: after a successful build, successful
host-side tests, and successful draft PR publication, the command still failed
hard when auto-merge was denied by token permissions.

Created GitHub issue `#15` to fix that behavior and implemented the hardening
on `main`:

- `src/openclawcode/app/run-issue.ts` now catches merge errors after approval
  and keeps the run in `ready-for-human-review`
- the run history now records an actionable note when auto-merge fails
- permission-related merge failures now explicitly advise that
  `GH_TOKEN` / `GITHUB_TOKEN` needs pull-request and contents write access
- `src/openclawcode/testing/run-issue.test.ts` now covers the merge-failure
  path and asserts the run remains usable

Validated the hardening locally with:

- `pnpm exec vitest run src/openclawcode/testing/run-issue.test.ts --pool threads`
- `pnpm exec vitest run --config vitest.openclawcode.config.mjs`

Then reran the real issue `#13` workflow on a fresh branch with both
`--open-pr` and `--merge-on-approve` enabled:

- `node scripts/run-node.mjs code run --issue 13 --owner zhyongrui --repo openclawcode --repo-root /home/zyr/pros/openclawcode --branch-name openclawcode/issue-13-stage-label-retry --builder-agent main --verifier-agent main --test "pnpm exec vitest run --config vitest.openclawcode.config.mjs" --open-pr --merge-on-approve --json`

Observed result after the hardening:

- run record `zhyongrui-openclawcode-13-1773113108387.json`
- builder again produced the command-layer change and opened real draft PR
  `https://github.com/zhyongrui/openclawcode/pull/16`
- verifier approved the run for human review
- auto-merge still failed because the current token cannot call the merge API
- unlike the first attempt, the command now returned a final approved run JSON
  instead of crashing
- run history recorded:
  `Auto-merge failed: GitHub token cannot merge pull requests. Ensure GH_TOKEN/GITHUB_TOKEN has pull request and contents write access. ...`

This confirmed two things:

- the real merge API is currently blocked by token scope, not by the workflow
  orchestration itself
- `openclawcode` now degrades correctly when that final merge step cannot be
  completed
