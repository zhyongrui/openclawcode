# Development Log - 2026-03-10

## Goal

Stabilize real GitHub issue execution so host-side validation commands can run
inside isolated issue worktrees without reinstalling dependencies for every run.

## Problem Observed

Real validation of issue `#4` proved that the workflow itself was behaving
correctly, but host-side test execution failed after the builder finished:

- the builder correctly detected that issue `#4` was already implemented
- the host then ran `pnpm exec vitest run --config vitest.openclawcode.config.mjs`
  inside the issue worktree
- that worktree had no local `node_modules`, so `pnpm exec vitest` failed with
  `Command "vitest" not found`

This was a worktree-host integration bug, not an issue-classification or prompt
targeting bug.

## Implementation

Added shared install artifact linking to the git worktree manager:

- `src/openclawcode/worktree/manager.ts` now defines
  `SHARED_INSTALL_ARTIFACTS = ["node_modules"]`
- when preparing a new or reused issue worktree, the manager now symlinks
  `repoRoot/node_modules` into the worktree if the source exists and the target
  does not
- changed-file collection now ignores shared install artifacts so the symlinked
  `node_modules` tree does not appear as workflow output noise

Added regression coverage in `src/openclawcode/testing/worktree.test.ts`:

- temp repos now include a fake `node_modules/.bin/vitest`
- new coverage proves the worktree receives a symbolic-link `node_modules`
  pointing back to the repository root
- changed-file reporting still excludes runtime/install artifacts

## Auto-Merge Classification Hardening After Issue #25

Ran the full real workflow against GitHub issue `#25`
(`verificationFollowUpCount`) with both draft-PR publication and

- `node scripts/run-node.mjs code run --issue 25 --owner zhyongrui --repo openclawcode --repo-root /home/zyr/pros/openclawcode --branch-name openclawcode/issue-25-verification-followup-count --builder-agent main --verifier-agent main --test "pnpm exec vitest run --config vitest.openclawcode.config.mjs" --open-pr --merge-on-approve --json`

Observed result:

- run record `zhyongrui-openclawcode-25-1773119460077.json`
- builder made the requested command-layer change and opened real draft PR
  `https://github.com/zhyongrui/openclawcode/pull/26`
- verifier approved the implementation for human review
- the run still stopped at `ready-for-human-review` because auto-merge was
  skipped under the non-command-layer policy path

Root cause:

- the scope classifier marked this issue as `mixed` even though the change was
  a narrow `openclaw code run --json` output slice
- the real issue wording legitimately mentioned `verification` and `verifier`,
  which added enough workflow-core score to trip the conservative
  `strong command issue` branch
- this made an otherwise safe command-layer run miss the auto-merge path

Implemented the hardening on `main` by:

- updating `src/openclawcode/roles/scope.ts` so strong command issues stay in
  `command-layer` when command signals clearly dominate workflow-core hints,
  instead of treating any moderate workflow hint count as automatically mixed
- extending `src/openclawcode/roles/scope.test.ts` with a regression test based
  on the real issue `#25` wording

Validated the classifier hardening locally with:

- `pnpm exec vitest run src/openclawcode/roles/scope.test.ts --pool threads`
- `pnpm exec vitest run --config vitest.openclawcode.config.mjs`

commit `d0862f2243` (`feat: implement issue #25`).
