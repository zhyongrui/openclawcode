## OpenClaw ChatOps Integration Direction

After reviewing the current OpenClaw code, the plugin/runtime boundary is good
enough to implement chat-driven issue execution with minimal core changes.

Important confirmed seams:

- plugin HTTP routes already exist for webhook ingress
- plugin services already exist for background workers
- plugin commands already exist for chat approval actions
- plugin runtime already exposes subagent/system helpers

That means the next practical integration path is:

1. GitHub issue webhook enters through a plugin HTTP route
2. adapter filters the event by repo and labels
3. adapter posts a chat approval prompt
4. user replies with a deterministic command such as `/occode-start`
5. adapter converts that command into an `openclawcode` run request
6. workflow core executes and adapter posts status updates back to chat

To lock this boundary down before wiring runtime registration, added a first
pure-function adapter slice in `src/integrations/openclaw-plugin/` covering:

- webhook intake decisions
- chat command parsing
- workflow run-request shaping
- status-message formatting

And added focused regression coverage in:

- `src/openclawcode/testing/openclaw-plugin-integration.test.ts`

## Atomic Approval-To-Queue Promotion

Hardened the approval flow so `/occode-start` no longer performs a
read-delete-write sequence across pending approvals and queue state.

Before this change, the path was:

1. read pending approval
2. delete pending approval
3. enqueue run

That left a small failure window where a crash or write error could lose the
pending approval without successfully enqueuing the issue.

Now the store performs this as one persisted transition:

- `promotePendingApprovalToQueue(...)`

Added store coverage for:

- atomic promotion from pending approval to queue
- direct promotion to queue when no pending approval exists
